Appendix K — B.5 — Overleaf + the AEA LaTeX Template

Your capstone paper has to look like an economics paper, because that is the form a referee — and the Schar Young Scholars Journal — expects to read. The tool the field uses to produce that look is LaTeX, and this section teaches you enough of it to compile the minimal-but-complete paper skeleton you met in Lab 8, by two routes: Overleaf, a browser editor that needs nothing installed, and a local latexmk/tectonic build on the camp container. It closes the loop you have been building all camp by wiring the paper’s compile step into the one-click replication packet of Lab 8, so that “rebuild the paper from raw data” is one command and the PDF is never hand-assembled.

A clarification up front, so you do not go looking for a tool we do not use: Lab 8 deliberately builds the skeleton on the plain article document class and notes that the official AEA class drops in with a one-line change. This appendix does the same. You learn the shape on article — which compiles anywhere, including a fresh Overleaf project with nothing extra installed — and swap to the AEA class only when the camp distributes the template file. The exact AEA class and BibTeX-style filenames are tagged [CHECK] throughout, because they belong to a file the camp hands you, not one we can copy here.

B.5.1 What LaTeX is, and why papers use it

You have written documents in a word processor: you type, and what you see on screen is what prints. LaTeX works differently. It is a typesetting system — you write a plain-text source file marked up with commands (\section{...}, \cite{...}, $\hat{\beta}$), and a program compiles that source into a finished, beautifully laid-out PDF. The model is “describe what you mean, let the system handle how it looks,” the same separation of content from presentation you have seen between data and a plot, or between a CSV and the table it becomes. You do not drag a figure into position or nudge an equation by hand; you say \includegraphics or open $$...$$ and LaTeX places it according to professional typesetting rules built in decades ago.

Why does empirical economics use it almost universally, when word processors exist? Four reasons, each of which matters for your paper specifically.

The first is mathematics. A regression equation, a matrix expression like \hat{\boldsymbol{\beta}} = (\mathbf{X}'\mathbf{X})^{-1}\mathbf{X}'\mathbf{y}, a properly set \xrightarrow{p} — LaTeX renders these the way the field reads them, and it is the source of the very math notation CONVENTIONS §3 standardizes for this book. A word processor’s equation editor is a struggle by comparison.

The second, and the one that matters most for the replication discipline you built in Lab 8, is that LaTeX source is plain text, which means it is programmable and diffable. Your analysis code can write a .tex table file directly (you saw 03_analysis.py do exactly this in Lab 8), and your paper can \input it — so the coefficient −1.8 lives in one generated file and flows into the PDF untouched by human hands. A binary word-processor document cannot be generated by a script or version-controlled line-by-line; a .tex file can be committed to Git, reviewed in a pull request, and rebuilt from raw data on a stranger’s machine. LaTeX is what makes the “no number is ever typed by hand” promise mechanically enforceable.

The third is cross-references and citations that maintain themselves. You label a table \label{tab:main} and refer to it as Table~\ref{tab:main}; if you reorder sections and it becomes Table 3, every reference updates on the next compile. You cite a paper with \cite{key} and LaTeX assembles the reference list and numbers it for you. Renumbering by hand — the classic way to ship a paper that says “see Table 2” pointing at Table 4 — simply does not happen.

The fourth is consistency and journal conformance: every section heading, every caption, every margin is set by the document class, so the paper looks uniform and matches the journal’s house style by loading the journal’s class file. That is precisely what the AEA template does.

B.5.2 Overleaf basics: projects and compiling

Overleaf is LaTeX in a web browser. It needs nothing on your machine — no TeX install, no command line — which is exactly why Lab 8 names it the fallback for students whose laptops have no LaTeX engine. You write source in the left pane, click Recompile, and the PDF renders in the right pane. For a camp where some students are on locked-down school laptops, this is the path that always works.

The unit of work in Overleaf is a project — a folder of files (your main.tex, a tables/ subfolder, a figures/ subfolder, a references.bib) that compile together into one PDF. To start your capstone paper, create a new blank project, then upload your paper/ folder preserving its structure: main.tex at the top, the tables/ and figures/ subfolders beneath it. The structure matters because your source refers to those subfolders by relative path — \input{tables/main_results.tex} and \graphicspath{{figures/}} only resolve if Overleaf keeps the folders where your .tex expects them, and it does.

Three settings get a first compile working, all under the project menu:

  • Main document. Tell Overleaf which file is the root to compile — main.tex. (Overleaf usually auto-detects the file containing \documentclass, but set it explicitly if it guesses wrong.)
  • Compiler. Set it to pdfLaTeX for the plain-article skeleton. When you add BibTeX citations (B.5.4), Overleaf still runs the right sequence automatically; you do not change the compiler for that.
  • Recompile. Click it. The PDF appears; warnings and errors show in a log panel you can expand. Overleaf reruns LaTeX as many passes as your cross-references and citations need, so a \ref that shows as ?? on a first manual pass resolves on its own here.

There is one discipline Overleaf does not enforce, and it is the same one Lab 8 hammers: the tables/ and figures/ you upload must be the ones your code just generated, not files you edited in the browser. Overleaf is a compiler, not a place to author results. If you fix a coefficient by hand in the Overleaf editor, you have broken the chain from raw data to PDF and reintroduced exactly the drift the replication packet exists to prevent. The honest workflow is: regenerate the artifacts with your analysis (locally or on the container), then upload them; Overleaf only typesets. Overleaf also supports a Git/GitHub link, so an advanced workflow pushes the generated paper/ folder from your repo and Overleaf pulls it — but the rule is unchanged: results are authored by code, Overleaf merely renders them.

B.5.3 A minimal AEA-style article skeleton

Here is the complete skeleton — the same one Lab 8 builds, reproduced so this appendix stands on its own as the setup reference. It is small but complete in shape: a title block, an abstract, the standard empirical-paper sections, a results section that pulls in a generated table and a generated figure, and a reference list. Write it to paper/main.tex.

\documentclass[12pt]{article}
\usepackage[margin=1in]{geometry}
\usepackage{graphicx}      % \includegraphics for figures
\usepackage{booktabs}      % \toprule/\midrule/\bottomrule for clean tables (Appendix D)
\usepackage{amsmath}       % aligned equations, \text, matrices
\usepackage{hyperref}      % clickable \ref and \cite
\graphicspath{{figures/}}  % so \includegraphics finds figures/ without a path prefix

\title{Fair-Lending Examination Programs and the Minority--White Denial Gap}
\author{Maya R.\thanks{NextGen FinTech Scholars, George Mason University.}}
\date{August 15, 2026}

\begin{document}
\maketitle

\begin{abstract}
We estimate how state fair-lending examination programs affect the county-level
minority--white mortgage-denial gap, exploiting the staggered timing of adoption.
We find the gap narrows by about 1.8 percentage points after adoption, with flat

B.5.4 BibTeX hygiene, and the natbib trap from Lab 8

The skeleton above hard-codes its one reference inside a thebibliography block. That is fine for a single citation, but a real paper cites twenty or fifty, and you do not want to format each by hand or renumber them when you add one. The scalable mechanism is BibTeX: you keep your references in a separate .bib database, cite each by a short key in the text, and a .bst bibliography style file formats the reference list automatically in the journal’s house style.

A .bib file is a plain-text list of entries, one per source. Put yours in paper/references.bib:

@article{callaway2021,
  author  = {Callaway, Brantly and Sant'Anna, Pedro H. C.},
  title   = {Difference-in-Differences with Multiple Time Periods},
  journal = {Journal of Econometrics},
  year    = {2021},
  volume  = {225},
  number  = {2},
  pages   = {200--230}
}

The token right after the brace — callaway2021 — is the citation key, your handle for this source. In the text you write \cite{callaway2021} (or \citet/\citep under natbib, below), and at the spot you want the reference list you replace the manual thebibliography block with two lines naming the style and the database:

\bibliographystyle{aer}          % the AEA/AER .bst style -- [CHECK] exact filename
\bibliography{references}        % reads references.bib; the .bib extension is implied

Now LaTeX builds the reference list from only the entries you actually \cite, formats them per the style, and numbers and orders them for you. Add a source by adding a .bib entry and citing it; delete one by removing the \cite. This is the version of “the data lives in one place” applied to your bibliography.

The natbib trap — the exact bug that bit the verification of Lab 8. There are two ways to make references, and you must pick one, never half of each. (1) The manual way: \cite{key} with a hand-written \begin{thebibliography}...\bibitem{key}...\end{thebibliography} block, as in the B.5.3 skeleton. (2) The BibTeX way: a .bib file with \bibliographystyle{} and \bibliography{}, as just shown. The mistake is to load the natbib package (which gives author-year citations like “Callaway and Sant’Anna (2021)” via \citet/\citep) and keep a manual thebibliography block. In author-year mode, natbib is incompatible with a hand-written thebibliography, and the build dies with Package natbib Error: Bibliography not compatible with author-year citations. The fix is to commit to one mechanism: either plain \cite{} + \bibitem{} with no natbib (the working skeleton), or full BibTeX with a .bib file and a .bst style (and then natbib’s \citet/\citep are fine, because the reference list is generated, not hand-written). Do not mix the half-and-half.

So natbib vs. plain, decided: plain \cite{} produces numbered references (“[1]”) and pairs happily with a manual thebibliography for a tiny paper. natbib produces the author-year citations economics readers expect (“Callaway and Sant’Anna, 2021”) and is what the AEA style assumes — but only with a generated BibTeX bibliography, never a hand-written one. For your capstone, the clean choice is the BibTeX path with the AEA .bst; the manual path in the skeleton is the training-wheels version that compiles with zero extra files. One more hygiene rule: every .bib entry is a citation, so every entry must be verifiable — CONVENTIONS §6 demands a real primary source for every empirical claim, and Chapter 6.5 warns that an LLM will fabricate perfectly formatted but nonexistent references. Verify each entry against the journal or a DOI before it enters references.bib, and tag anything you cannot confirm [CHECK] rather than letting fiction into your bibliography.

## B.5.1 What LaTeX is, and why papers use it

You have written documents in a word processor: you type, and what you see on screen is what prints. LaTeX works differently. It is a **typesetting system** — you write a plain-text *source* file marked up with commands (`\section{...}`, `\cite{...}`, `$\hat{\beta}$`), and a program *compiles* that source into a finished, beautifully laid-out PDF. The model is "describe what you mean, let the system handle how it looks," the same separation of content from presentation you have seen between data and a plot, or between a CSV and the table it becomes. You do not drag a figure into position or nudge an equation by hand; you say `\includegraphics` or open `$$...$$` and LaTeX places it according to professional typesetting rules built in decades ago.

Why does empirical economics use it almost universally, when word processors exist? Four reasons, each of which matters for *your* paper specifically.

The first is **mathematics**. A regression equation, a matrix expression like $\hat{\boldsymbol{\beta}} = (\mathbf{X}'\mathbf{X})^{-1}\mathbf{X}'\mathbf{y}$, a properly set $\xrightarrow{p}$ — LaTeX renders these the way the field reads them, and it is the source of the very math notation CONVENTIONS §3 standardizes for this book. A word processor's equation editor is a struggle by comparison.

The second, and the one that matters most for the replication discipline you built in Lab 8, is that **LaTeX source is plain text**, which means it is *programmable* and *diffable*. Your analysis code can *write* a `.tex` table file directly (you saw `03_analysis.py` do exactly this in Lab 8), and your paper can `\input` it — so the coefficient `−1.8` lives in one generated file and flows into the PDF untouched by human hands. A binary word-processor document cannot be generated by a script or version-controlled line-by-line; a `.tex` file can be committed to Git, reviewed in a pull request, and rebuilt from raw data on a stranger's machine. LaTeX is what makes the "no number is ever typed by hand" promise mechanically enforceable.

The third is **cross-references and citations that maintain themselves**. You label a table `\label{tab:main}` and refer to it as `Table~\ref{tab:main}`; if you reorder sections and it becomes Table 3, every reference updates on the next compile. You cite a paper with `\cite{key}` and LaTeX assembles the reference list and numbers it for you. Renumbering by hand — the classic way to ship a paper that says "see Table 2" pointing at Table 4 — simply does not happen.

The fourth is **consistency and journal conformance**: every section heading, every caption, every margin is set by the document class, so the paper looks uniform and matches the journal's house style by loading the journal's class file. That is precisely what the AEA template does.

## B.5.2 Overleaf basics: projects and compiling

**Overleaf** is LaTeX in a web browser. It needs nothing on your machine — no TeX install, no command line — which is exactly why Lab 8 names it the fallback for students whose laptops have no LaTeX engine. You write source in the left pane, click **Recompile**, and the PDF renders in the right pane. For a camp where some students are on locked-down school laptops, this is the path that always works.

The unit of work in Overleaf is a **project** — a folder of files (your `main.tex`, a `tables/` subfolder, a `figures/` subfolder, a `references.bib`) that compile together into one PDF. To start your capstone paper, create a new blank project, then **upload your `paper/` folder preserving its structure**: `main.tex` at the top, the `tables/` and `figures/` subfolders beneath it. The structure matters because your source refers to those subfolders by relative path — `\input{tables/main_results.tex}` and `\graphicspath{{figures/}}` only resolve if Overleaf keeps the folders where your `.tex` expects them, and it does.

Three settings get a first compile working, all under the project menu:

- **Main document.** Tell Overleaf which file is the root to compile — `main.tex`. (Overleaf usually auto-detects the file containing `\documentclass`, but set it explicitly if it guesses wrong.)
- **Compiler.** Set it to **pdfLaTeX** for the plain-`article` skeleton. When you add BibTeX citations (B.5.4), Overleaf still runs the right sequence automatically; you do not change the compiler for that.
- **Recompile.** Click it. The PDF appears; warnings and errors show in a log panel you can expand. Overleaf reruns LaTeX as many passes as your cross-references and citations need, so a `\ref` that shows as `??` on a first manual pass resolves on its own here.

There is one discipline Overleaf does *not* enforce, and it is the same one Lab 8 hammers: **the `tables/` and `figures/` you upload must be the ones your code just generated, not files you edited in the browser.** Overleaf is a *compiler*, not a place to author results. If you fix a coefficient by hand in the Overleaf editor, you have broken the chain from raw data to PDF and reintroduced exactly the drift the replication packet exists to prevent. The honest workflow is: regenerate the artifacts with your analysis (locally or on the container), then upload them; Overleaf only typesets. Overleaf also supports a Git/GitHub link, so an advanced workflow pushes the generated `paper/` folder from your repo and Overleaf pulls it — but the rule is unchanged: results are authored by code, Overleaf merely renders them.


Because `paper` *depends on* `tables figures`, `make` guarantees the generated `.tex` table and the `.pdf` figure exist *before* LaTeX runs — so `\input{tables/main_results.tex}` and `\includegraphics{event_study.pdf}` always find current, code-written files, never stale or hand-edited ones. For a teammate without `make` (Windows laptops especially), Lab 8's `run_all.py` does the same thing and gates the compile on whether an engine exists, via `shutil.which`: if `latexmk` or `tectonic` is present it compiles; if neither is, it skips cleanly and prints that the `.tex` is assembled and ready to build on Overleaf. That gate is the bridge between this section's two roads — local build when an engine exists, Overleaf when it does not — wired so the *same* packet works on both.

The honesty test from Lab 8 closes the loop: `make clean && make all` deletes every derived file and rebuilds from nothing, and `md5sum -c before.txt` confirms the table `.tex` comes back byte-identical. (The figure PDF comes back identical too *only if* you pinned its metadata date — `savefig(..., metadata={"CreationDate": None})` — because matplotlib otherwise stamps the wall-clock time into the PDF; knowing that a changed timestamp is cosmetic while a changed coefficient is a broken packet is the skill.) The LaTeX you learned here is the typesetting half of that machine: code writes the numbers into `.tex`, LaTeX renders the `.tex` into a PDF, and `make` chains the two so no number is ever typed by hand.
